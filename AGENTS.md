# AGENTS.md

## Local Agent Guide

This file provides local operating instructions for AI coding agents working in this repository.

It is intended for local use only and should not be committed unless explicitly requested.

## Project Context

This repository contains Cubby, a portable AI workflow framework for K-5 educators, special education teachers, instructional coaches, BCBAs, interventionists, and related classroom-support professionals.

Cubby installs structured AI workflows into teacher workspaces. It is not itself an LMS, SIS, IEP system, grading system, or classroom management platform.

Use PLAN.md as the source of truth for product requirements, architecture, milestones, and implementation priorities.

## How to Work in This Repository

Before making changes:

1. Read PLAN.md.
2. Inspect the current repository structure.
3. Determine what already exists.
4. Make the smallest coherent change that advances the requested task.
5. Preserve existing local work unless explicitly told to replace it.

Do not assume the repository already matches PLAN.md. Verify first.

## Implementation Style

Prefer:

* Simple TypeScript
* Small modules
* Clear names
* Explicit file paths
* Deterministic behavior
* Readable Markdown, YAML, and JSON
* Conservative filesystem writes
* Useful CLI output
* Tests for file creation, validation, and overwrite behavior

Avoid:

* Clever abstractions before they are needed
* Silent overwrites
* Large rewrites unrelated to the current task
* Unrequested integrations
* Hardcoded absolute paths
* Generated files that are difficult to inspect
* Mixing framework source files with installed workspace output

## Source of Truth

Use these files as guidance in this order:

1. User's explicit instruction in the current task
2. PLAN.md
3. Existing repository code and tests
4. This AGENTS.md

When instructions conflict, prefer the user's current task. If ambiguity remains, choose the smallest safe implementation and document the assumption.

## Repository vs Installed Workspace

Keep this distinction clear:

* The repository contains Cubby source code, CLI logic, templates, schemas, workflows, agents, rules, validators, profiles, and adapters.
* An installed workspace is generated by Cubby for a teacher or education-support user.

Do not confuse repository implementation files with files generated into an installed workspace.

Generated workspace files should be treated as scaffolded artifacts. Framework source files should remain platform-neutral unless they live under an adapter-specific path.

## Platform Independence

Cubby should remain provider-independent at the shared-source level.

Provider-neutral content belongs in areas such as:

```text
src/agents/
src/workflows/
src/commands/
src/rules/
src/templates/
src/schemas/
src/validators/
src/hooks/
src/profiles/
```

Provider-specific behavior belongs under adapter-specific paths such as:

```text
src/adapters/<adapter-name>/
```

Do not add provider-specific assumptions to shared framework files unless the task explicitly requires it.

## Filesystem Safety

Be conservative with all file writes.

When generating or updating files:

* Do not overwrite user-authored files unless explicitly instructed.
* Preserve local customization directories.
* Prefer create-if-missing behavior for local files.
* Use managed-file metadata where the framework requires it.
* If a managed file appears locally edited, preserve it and report the issue instead of overwriting.
* Keep generated output deterministic when possible.

Local customization, output, export, and log directories should be treated as user-owned.

## Validation and Tests

When changing behavior, update or add tests where practical.

Prefer tests that verify observable behavior:

* CLI commands run successfully
* expected files are created
* expected directories are created
* malformed state/config files fail validation
* repeat runs do not overwrite local files
* manifests or metadata accurately describe generated files
* generated workspace structure matches expectations

After changes, run the relevant checks available in the repo, such as:

```text
npm run check
npm run build
npm test
```

If a command is unavailable, report that plainly.

## CLI Behavior

CLI behavior should be predictable and inspectable.

Prefer CLI output that clearly reports:

* what was created
* what was skipped
* what was updated
* what was preserved
* what failed
* what warnings require attention

Do not hide important filesystem decisions.

## Documentation Style

Documentation should be clear, practical, and teacher-centered.

The product tone should feel:

* warm
* calm
* professional
* useful
* respectful of educator judgment

Avoid making Cubby sound like an enterprise compliance appliance. Also avoid making it childish.

## Education Domain Guardrails

Cubby is designed for education workflows that may involve sensitive student contexts.

When authoring framework guidance, templates, validators, or workflow instructions:

* Preserve teacher and qualified-professional authority.
* Use strengths-based language.
* Avoid deficit framing.
* Avoid unnecessary student-identifying information.
* Distinguish observed data from interpretation.
* Require human review for sensitive student-specific outputs.
* Treat parent/family communication, IEP-adjacent content, behavior-support content, and progress interpretation as review-gated.

Do not create framework content that implies AI can replace professional educational judgment.

## Adapter Guidance

Adapters translate shared Cubby concepts into a specific platform's expected files or conventions.

Adapters should not own the core product model. They should map shared agents, workflows, commands, rules, templates, schemas, validators, and profiles into provider-specific formats.

When working on adapters:

* Keep adapter code thin.
* Document adapter-specific behavior.
* Avoid duplicating shared rules unnecessarily.
* Keep generated files readable.
* Make generated output easy to diff.

## Dependency Guidance

Prefer minimal dependencies.

Before adding a dependency:

1. Check whether the repo already has a suitable utility.
2. Prefer established, boring packages.
3. Avoid large frameworks for small CLI tasks.
4. Document why the dependency is useful if it is not obvious.

## Error Handling

Errors should be understandable and actionable.

Prefer messages that explain:

* what failed
* which file or path was involved
* what the user or developer can do next

Avoid vague errors.

## When Unsure

When a decision is ambiguous:

1. Re-read PLAN.md.
2. Inspect existing implementation patterns.
3. Choose the smallest safe change.
4. Preserve local/user-authored files.
5. Add a TODO only when it is specific and actionable.
6. Summarize the assumption in the final response.

Do not invent large product behavior that is not requested or described in PLAN.md.

## Final Response Expectations

When finishing a task, summarize:

* what changed
* which files were touched
* which checks were run
* whether anything failed or was skipped
* any follow-up work that remains

Be concise but specific.
